# About the Implementation: What We Planned vs What Actually Happened

This document captures the lessons learned during the implementation of Project Capacitor — the gaps between what the design assumed and what reality required. It covers four categories: Prisma v7 breaking changes, Docker containerization surprises, ESM module friction, and architectural decisions that were revised under time pressure.

The value of a postmortem isn't in cataloging failures — it's in understanding *why* the plan diverged from reality, so the next plan is better calibrated.

## Prisma v7: The Ground Shifted Under the Schema

The implementation spec and ADR-001 were written assuming Prisma v5/v6 patterns, which dominate online guides and LLM training data. Prisma v7 (7.8.0) was installed by `npm install prisma` and introduced several breaking changes that surfaced at runtime, not at planning time.

### What changed

**The `datasource.url` field was removed from `schema.prisma`.** In Prisma v5/v6, the database connection string lived in the schema file: `url = env("DATABASE_URL")`. In v7, this is handled exclusively by `prisma.config.ts`, a new configuration file that Prisma generates during `prisma init`. The spec's schema was written with the `url` field, and the first `prisma db push` would have failed if the subagent implementing Task 1 hadn't noticed and adapted.

**`PrismaClient` now requires a driver adapter.** A plain `new PrismaClient()` with no arguments throws `PrismaClientConstructorValidationError` in v7. The client must be constructed with a driver adapter — in our case, `PrismaPg` from `@prisma/adapter-pg`, which wraps the `pg` PostgreSQL driver. This meant installing two additional packages (`@prisma/adapter-pg`, `pg`) and rewriting the Prisma client singleton to pass the adapter.

**The seed configuration moved.** The spec assumed `"prisma": { "seed": "..." }` in `package.json`. Prisma v7 reads seed configuration from `prisma.config.ts` under `migrations.seed`. The `package.json` key is ignored. This was discovered when `npx prisma db seed` silently did nothing.

**Node.js >= 22 is required.** The `@prisma/streams-local` internal package declares `engines: { node: ">=22.0.0" }`. Our Dockerfiles initially used `node:20-alpine`, which caused `npm ci` to emit warnings and Prisma operations to fail at runtime.

### Why this matters

Prisma v7 was a major version release that happened between when the LLM (Gemini, in the design discussion) generated its code examples and when we actually installed the package. The Gemini conversation's Prisma schema, seed script, and client setup were all v5/v6 patterns. Every one of them needed adaptation.

This is a general risk with AI-assisted development: **code suggestions reflect the training data's version of a library, not the installed version.** The mitigation is to always run `npx prisma init` and read the generated files rather than copying schema patterns from LLM output or documentation that may be stale.

### What we'd do differently

Pin the Prisma version in the spec (`prisma@^5.0.0`) or note that the code examples assume v5 patterns and may need adaptation. Better: scaffold first, read the generated files, then write the spec based on what's actually installed.

## Docker: The Gap Between "It Works Locally" and "It Works in a Container"

Three Docker issues surfaced during `docker-compose up --build`, each revealing a different category of assumption.

### The npm lockfile format incompatibility

The lockfiles (`package-lock.json`) were generated locally with Node 24 / npm 11. The Dockerfiles used `node:20-alpine` (npm 10). `npm ci` — which requires an exact lockfile match — failed because the lockfile format from npm 11 is incompatible with npm 10.

This is a subtle trap: `npm ci` is the recommended Docker best practice (deterministic installs), but it assumes the lockfile was generated by the same npm version running in the container. When the developer's local Node version is ahead of the Docker image, `npm ci` breaks silently.

The fix was twofold: upgrade to `node:22-alpine` (for Prisma compatibility) and switch from `npm ci` to `npm install` (more forgiving across npm versions). In a production setting, the proper fix would be to ensure the Docker image's Node version matches the team's local development version, or to generate the lockfile inside the container.

### The port conflict

Port 5432 was already occupied by another PostgreSQL container (`onecli-postgres-1`) from an unrelated project. Docker Compose doesn't detect port conflicts until container startup, and the error message ("address already in use") doesn't identify which process holds the port.

The fix was to publish on port 5433 instead (`5433:5432`). But this created a split: locally, the backend connects to `localhost:5433`, while in Docker, the backend connects to `db:5432` (the container's internal port). The `DATABASE_URL` in `docker-compose.yml` must use the internal port, while the `.env` for local development uses the external port. This split is easy to get wrong and is a common source of "works in Docker, breaks locally" bugs.

### The tsconfig scope leak

`tsc` (TypeScript compiler) picks up all `.ts` files matching the `include` pattern. Without an explicit `include`, it defaults to `**/*`, which includes `prisma.config.ts` and `prisma/seed.ts` — both outside `rootDir: "src"`. Locally this worked because `tsx` (used for dev mode) ignores `rootDir` constraints. But `tsc` (used for the Docker production build) enforces them strictly.

The fix was adding `"include": ["src"]` and `"exclude": ["node_modules", "dist", "prisma"]` to `tsconfig.json`. This is a case where the dev workflow (`tsx`) masked a build workflow (`tsc`) problem — the kind of issue that only surfaces in CI/CD or Docker builds.

## ESM Modules: Death by a Thousand Import Errors

The backend uses `"type": "module"` in `package.json`, enabling ECMAScript modules. This choice was made by the Task 3 subagent to resolve a conflict between `verbatimModuleSyntax: true` in tsconfig and CommonJS `require()`. While ESM is the future of Node.js, it introduced persistent friction throughout implementation.

### The `.js` extension requirement

Every import in ESM must include the file extension: `import prisma from '../lib/prisma.js'`, not `../lib/prisma`. This is counterintuitive because the source files are `.ts`, not `.js` — the `.js` refers to the compiled output. Every subagent implementing a new file had to remember this, and several forgot on first attempt.

### The `import type` requirement

With `verbatimModuleSyntax: true`, TypeScript requires that type-only imports use the `import type` syntax. `import { Request, Response } from 'express'` fails at runtime in Node 24's strict ESM mode because Express doesn't export those as runtime values — they're type declarations only. The fix is `import type { Request, Response } from 'express'`.

This issue appeared in two separate tasks (validation middleware and task service) and required the same fix each time. The pattern is predictable: any import from `@types/*` packages must use `import type`.

### Why ESM was still the right choice

Despite the friction, ESM was correct for this project. The alternative — CommonJS with `verbatimModuleSyntax: false` — would have required disabling a TypeScript strictness feature that catches real bugs. And the ecosystem is moving toward ESM: Prisma v7, Vite, and the Vercel AI SDK all work best with ESM. The friction is real but temporary — as the Node.js ecosystem completes its ESM migration, these issues will become less common.

## Architectural Decisions Revised Under Pressure

The devil's advocate analysis (wiki/notes/devils-advocate-blind-spots.md) identified five blind spots before implementation began. Three of them directly shaped what was actually built.

### Explicit join tables → Implicit M:N

ADR-001 originally specified explicit join tables (`DeveloperSkill`, `TaskSkill`) for "future extensibility." The devil's advocate called this YAGNI — and was right. The implementation used Prisma's implicit many-to-many (`skills Skill[]` on both models), which eliminated boilerplate in every query touching skills. The "future extensibility" of explicit tables can be added in 10 minutes if proficiency metadata is ever needed.

The lesson: in a time-boxed project, "future-proof" abstractions are a liability. They add complexity today for a benefit that may never materialize.

### Three-state FSM with transition guards → Free transitions

The ambiguity solutions document proposed blocking the TODO → DONE transition (must pass through IN_PROGRESS). The devil's advocate pointed out this was an invented constraint that would frustrate evaluators testing with simple tasks. The implementation allows all transitions freely, with only one guard: entering DONE checks that all subtasks are DONE.

The lesson: constraints should come from the spec, not from architectural taste. Every invented rule is a potential user frustration point.

### Tree-table with lazy loading → Flat table with indent

The proposed Task List was an expandable tree-table with chevrons, URL-persisted expand state, and lazy subtask loading — estimated at 8-12 hours of frontend work. The devil's advocate flagged this as consuming half the frontend budget for a feature the spec's wireframe showed as a flat table.

The implementation used a flat table with CSS `padding-left` indentation and a "↳" prefix for subtasks. It took approximately 30 minutes to implement and satisfies the spec's requirement that "all existing Tasks and their relevant attributes should be listed." The recursive form on the Create page (the spec's actual requirement for recursive UI) demonstrated React recursive component competency.

The lesson: in a time-boxed evaluation, a clean simple solution that works beats an ambitious complex solution that might be buggy. The Create page's recursive `<TaskFormNode />` is where the evaluators see recursive React skill — the List page doesn't need to repeat it.

## The Meta-Lesson: Plans Are Hypotheses

The most important lesson isn't about any specific technology. It's that **an implementation plan is a hypothesis about what will work, not a guarantee.** The plan assumed Prisma v5 patterns, Node 20 compatibility, and `npm ci` in Docker. All three assumptions were wrong.

What made the implementation succeed despite wrong assumptions was:

1. **Subagents with autonomy to adapt.** Each task's implementer read the actual files, discovered the actual behavior, and adapted. They didn't blindly follow the plan — they followed the intent.

2. **Fail-fast discovery.** Issues surfaced immediately (Prisma errors on first run, Docker build failures on first build) rather than hiding until production.

3. **The wiki as institutional memory.** When Task 3's subagent discovered ESM was needed, that context was available to Task 4's subagent through the committed code. The wiki's decision log (ADRs) meant late tasks didn't repeat early mistakes.

The plan's value wasn't in being right — it was in being *specific enough to be wrong in discoverable ways.* A vague plan ("set up the database") would have hidden the Prisma v7 issues until much later. A specific plan ("write this exact schema, run this exact command") failed fast and visibly, making the adaptation quick.
